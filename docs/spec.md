# Bus Delays Dashboard — канонический стратегический документ

Единственный канонический документ проекта. Продуктовые, архитектурные и
дата-решения живут здесь; процессное состояние (статусы, вердикты, блокеры,
evidence) — только в `.agent/memory.db` через `memgraph` (см. `AGENTS.md`).

## 1. Что строим и зачем

Standalone-дашборд для анализа опозданий корпоративных развозок из
HR-файла `HR report.xlsx` (иврит). Стиль Power BI, пользователь —
нетехнический сотрудник HR. Двойная цель:

1. **Контроль подрядчика развозки** — какие маршруты/города опаздывают, как
   часто, каков масштаб (человеко-минуты).
2. **Расчёт зарплаты** — сколько времени докредитовать каждому сотруднику за
   опоздания по вине развозки (сотруднику зачитываются минуты опоздания);
   в drill-in видно реальное время прибытия для сверки.

## 2. Поставка: один HTML-файл (решение)

Каноническая поставка — **один файл `BusDelaysDashboard.html`**: открывается
любым современным браузером (Edge/Chrome), файл Excel выбирается один раз,
данные сохраняются в localStorage и дашборд дальше открывается сам.

Обоснование: корпоративная машина пользователя управляется политикой —
Windows Defender с Tamper Protection блокирует неподписанные exe и молча
игнорирует локальные исключения; это нечинимо без IT. Браузерный файл
политикой не блокируется.

**Non-goals (запрещённый объём):**

- никакого `.exe`/PyInstaller (путь закрыт 2026-07-02; Python-бэкенд с
  pytest-набором удалён из рабочего дерева, восстановим из git-истории до
  коммита `19e7c58`);
- никакого сервера, установки, интернет-зависимостей в рантайме;
- никаких фреймворков/бандлеров — vanilla HTML/CSS/JS + вендоренный SheetJS;
- `HR report.xlsx` (PII) никогда не коммитится (история от него очищена);
  данные пользователя не покидают его машину.

## 3. Контракт данных (`HR report.xlsx`)

Лист `Sheet1`, «широкая» таблица:

- строка 1 — даты над колонками «опоздание» каждого дня;
- строка 2 — заголовки на иврите; колонки ищутся по заголовкам, не по буквам:
  `מספר עובד` (номер сотрудника), `שם פרטי` (имя), `שם משפחה` (фамилия),
  `עיר` (город), `הסעה` (маршрут), `זמן איחור` (минуты опоздания),
  `זמן הגעה` (реальное время прибытия HH:MM);
- строки 3+ — по сотруднику на строку (полный ростер, meta-колонки);
- заполнены только опоздания (пустая ячейка = приехал вовремя);
- одно и то же опоздание проставлено каждому пассажиру маршрута (опоздал
  автобус, не человек);
- **значения маршрутов и городов — на иврите** (например
  `פתח תקווה` = Петах-Тиква, `לוד` = Лод); русские названия в старых записях —
  транслитерация, в файле их нет.

Семантика разбора (unpivot): одна запись на (сотрудник × день с опозданием):
`{employee_no, first_name, last_name, city, route, date(ISO), weekday,
late_min, arrival(HH:MM|null)}`. Неделя начинается с **воскресенья**
(израильская рабочая неделя вс–пт).

## 4. Приоритет данных и UX (утверждено пользователем)

Приоритет: **Города → Даты → Сотрудники**. Города — главный визуал (карточки);
даты — фильтр; сотрудники — третичны (drill-in для зарплаты). Никогда не
делать сотрудников видом по умолчанию.

- **KPI-полоса**: Cases, Total Late (min), Employees (весь ростер), Cities
  (все), Routes. Полноростерные цифры, не только «с опозданиями».
- **Слайсеры**: Sort by, Employee (автодополнение по фамилии), Date From/To,
  Clear. Фильтрует пользователь — не предфильтровывать за него.
- **Карточки городов**: ВСЕ города (в т.ч. без опозданий — нейтрально-приглушённые
  внизу, «no delays»), сортировка по убыванию суммарного опоздания; сетка
  выравнивается по количеству городов (15 → 5×3).
- **Кросс-фильтр в стиле Power BI**: клик по городу подсвечивает его, приглушает
  остальные (не удаляет!) и открывает drill-in.
- **Drill-in drawer** (выезжает справа): жители города из полного ростера,
  крупный номер сотрудника, имя, маршрут; по дням: дата · +мин · реальное
  время прибытия; жители без опозданий — приглушённо.
- **Первый запуск**: английская подсказка (“Change file…” один раз); после
  первого выбора файла данные живут в localStorage и дашборд открывается сам.
  [SUPERSEDED 2026-07-02: подсказка была по-русски — пользователь запретил
  русский текст в поставке.]
- **Export** текущего среза (Excel/CSV).
- Общий принцип: строить как знакомый инструмент (Power BI), не изобретать
  UI-паттерны; показывать полные данные.

## 5. Визуальный язык

Палитра Claude/Anthropic: тёплый кремовый фон `#F5F4EE`, акцент клэй `#CC785C`,
тёмно-тёплый хедер `#2B2924`; severity: зелёный `#5E8C61`, янтарный `#C7972F`,
красный `#B5482E`. Ярлыки UI — английские; значения данных — иврит с
`dir="auto"`.

**Язык поставки (правило пользователя, 2026-07-02): в поставке нет русского
текста.** Весь UI-текст, включая подсказки и сообщения об ошибках, — английский;
иврит допустим только как значения данных из исходного файла. Кириллица внутри
вендоренного SheetJS (таблицы кодировок) — внутренности библиотеки, не UI.

## 6. Архитектура и раскладка репозитория

Вся логика — в браузерном JS; исходники в `src/web/`, поставка собирается
в один файл.

- `src/web/standalone.html` — скелет страницы (ссылается на css и три скрипта);
- `src/web/engine.js` — движок данных, зеркало бывших excel_reader/filters/
  aggregate/api: `Engine.parseWorkbook(arrayBuffer) → {records, roster}`,
  `applyFilters`, `totals`, `aggregateBy(view)`, `citiesFull(roster, records)`
  (все города, 0 для беспроблемных), `buildDashboard`, `buildState`;
- `src/web/web-app.js` — UI-клей: рендер, кросс-фильтр, drawer, localStorage;
- `src/web/styles.css` — стили;
- `src/web/xlsx.full.min.js` — вендоренный SheetJS (парсит xlsx в браузере;
  разбор идёт с `raw:true` — без него контрольные числа не сходятся);
- `build.ps1` — инлайнит исходники в `BusDelaysDashboard.html`
  (чистая подстановка, круговая сборка байт-идентична);
- `BusDelaysDashboard.html` — собранная поставка, трекается в git;
- `memgraph/` + `.agent/memory.db` — процессная память (см. `AGENTS.md`;
  слои `memgraph/venv/`, `memgraph/vendor/`, `.env` — пер-машинные,
  в git не идут).
