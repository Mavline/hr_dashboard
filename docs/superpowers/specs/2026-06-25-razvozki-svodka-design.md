# Дизайн: дашборд опозданий развозок (десктоп-приложение)

- **Дата:** 2026-06-25
- **Статус:** на ревью (финальная форма — десктоп-приложение с постоянным путём)
- **Источник данных:** `HR report.xlsx` (табель опозданий служебной развозки, иврит)

## 1. Контекст и цель

Каждый период отдел кадров получает Excel-файл с фиксацией опозданий служебной
развозки (`הסעה`). Когда автобус маршрута опаздывает, опоздание (`זמן איחור`,
в минутах) проставляется **всем пассажирам этого маршрута одинаково** — это
опоздание автобуса, а не людей. Также фиксируется время прибытия (`זמן הגעה`).
Файл фиксирует только факты опозданий (приехавшие вовремя не отмечаются).

Инструмент решает **две задачи**:

1. **Начисление.** По каждому сотруднику — суммарное «лишнее» время за выбранный
   срез, которое нужно доплатить сверх зафиксированного.
2. **Контроль развозки.** По каждому маршруту — как часто опаздывает и какой
   масштаб проблемы создаёт (человеко-минуты потерь).

**Идея подхода.** Не строить расчётный движок, а развернуть данные в «длинную»
таблицу и дать **интерактивные фильтры по каждому полю + группировку + живые
итоги**. В длинном формате метрики вырождаются в суммы по отфильтрованным
строкам, поэтому «расчёт» = суммирование видимого.

## 2. Пользователь и сценарий

- **Пользователь:** сотрудник отдела кадров, не технический.
- **Частота:** умеренная.
- **Источник всегда один и тот же файл**, путь к нему задаётся один раз и
  запоминается.
- **Сценарий (одно действие):** двойной клик по ярлыку приложения → оно сразу
  читает файл по сохранённому пути и показывает дашборд. Никаких «открыть, потом
  перетащить» — шаг ровно один.
- При необходимости — кнопка «Сменить файл» (нативный диалог выбора), новый путь
  сохраняется.
- **Приватность:** всё локально, данные никуда не отправляются, интернет не нужен.
  Исходный файл не изменяется.

## 3. Запуск, путь к файлу и настройки

- Путь к источнику хранится в `config.json` (рядом с приложением).
- **При старте:** если путь сохранён и файл существует → читаем и показываем
  дашборд. Если пути ещё нет (первый запуск) или файл не найден → нативный диалог
  выбора файла; выбранный путь сохраняется.
- **Смена источника:** кнопка «Сменить файл» в интерфейсе → диалог → перечитать и
  перерисовать; путь обновляется в `config.json`.
- **Текущий путь виден** в шапке дашборда (чтобы было понятно, что открыто).
- Кнопка «Обновить» — перечитать тот же файл (если он изменился на диске).

## 4. Формат входного файла и разбор

Парсер **привязан к конкретному формату** этого файла (не универсальный). Лист с
данными (в текущем файле — `Sheet1`; `Sheet2` пустой и игнорируется):

- **Строка 1:** даты над колонкой «опоздание» каждого дня.
- **Строка 2:** заголовки колонок (иврит).
- **Строки 3+:** по одному сотруднику в строке.

Колонки метаданных сотрудника:

| Заголовок (иврит) | Поле |
|---|---|
| `מספר עובד` | Employee No |
| `שם פרטי` | First Name |
| `שם משפחה` | Last Name |
| `עיר` | City |
| `הסעה` | Route |

Далее — повторяющиеся **пары колонок по дням**: `זמן איחור` (опоздание, мин) и
`זמן הגעה` (время прибытия); дата дня берётся из строки 1 над колонкой опоздания.
Колонки без заголовка (разрозненные числа в `P` исходного файла) **игнорируются**.

Если структура не совпадает с ожидаемой (нет нужных заголовков) → понятное
сообщение, без сбоя.

## 5. Модель данных

**Внутреннее представление (движок, пользователь не видит).** Данные приводятся к
плоскому списку фактов `(сотрудник × день)` — по одной записи на каждое
зафиксированное опоздание:
`Employee No | First Name | Last Name | City | Route | Date | Weekday | Late (min) | Arrival`.
Это **разворот из «широкого» формата в «длинный» (unpivot)** — ключевой первый
шаг: столбцы-даты исходника не используются как поля, а превращаются в значения
поля `Date` (даты «растут вниз» строками, а не «вширь» столбцами).
На этом списке применяются фильтры и считаются любые агрегаты. Сумма `Late` по
набору записей = человеко-минуты среза; сумма по записям сотрудника = его доплата.

**Видимое представление (по умолчанию).** Дашборд по сути повторяет исходную
таблицу: строка = **сотрудник**, те же поля (таб.№, имя, фамилия, город,
маршрут), но **не в подневной форме** — поле опоздания показывает **агрегат за
текущий период** (по умолчанию — весь файл). Фильтры сужают набор фактов, и
агрегаты в строках **пересчитываются** под фильтр. Пользователь видит и фильтрует
«те же поля, что в исходнике, только с агрегированной информацией».

## 6. Дашборд: представление, фильтры, итоги

**Представление по умолчанию — по сотруднику** (суть исходной таблицы), сортировка
по Total Late ↓ — готовая ведомость к начислению:
`Employee No | First Name | Last Name | City | Route | Late Cases | Total Late (min) | Avg Late (min)`
Клик по строке раскрывает детали сотрудника по дням
(`Date | Weekday | Late (min) | Arrival`).

**Переключатель разреза** («View by»): Employee (по умолчанию) / Route / Date /
Week / Weekday. В разрезе Route строка = маршрут с теми же агрегатами
(`Route | Late Cases | Total Late (min) | Avg Late (min) | Employees`) — рейтинг
проблемности развозки; разрезы Date/Week/Weekday дают динамику.

**Фильтры по каждому полю** (комбинируются, пересчитывают агрегаты в строках):
- Employee (поиск по имени/фамилии/таб. номеру);
- City, Route — мультивыбор из списка;
- Date — диапазон «с — по» (это и есть выбор периода);
- Weekday — выбор дней (вс–пт).

**Живые итоги** (панель, пересчёт при любом фильтре): Cases · Total Late Minutes
(= человеко-минуты) · Employees affected · Routes affected · Days covered.

**Сортировка** — по клику на заголовок столбца.

**Экспорт** — кнопка «Скачать в Excel/CSV» выгружает текущий видимый срез (с
учётом разреза и фильтров) через нативный диалог сохранения.

**Внешний вид.** Современный аккуратный UI на HTML/CSS: шапка с названием и
текущим путём к файлу + кнопки («Сменить файл», «Обновить», «Экспорт»); панель
фильтров; строка карточек-итогов; основная таблица с зафиксированной шапкой,
зеброй строк и сортировкой по клику. Поддержка иврита/RTL в значениях
(`dir="auto"`). Перед реализацией основной логики делается HTML-макет дашборда и
согласуется с заказчиком.

## 7. Архитектура

Десктоп-приложение: **Python-бэкенд** (доступ к ФС, парсинг, экспорт) + **HTML/JS
дашборд** в окне webview. Разделение: Python отдаёт готовую длинную таблицу как
JSON; JS отвечает за фильтры/группировку/итоги/рендер.

Python (вся логика и доступ к данным — тестируется через `pytest`):
- **`config`** — чтение/запись `config.json` (путь к источнику).
- **`excel_reader`** — открыть файл по пути, разобрать под конкретный формат и
  развернуть в длинную таблицу (unpivot); чистая, тестируемая логика.
- **`filters`** — применение фильтров к списку записей (чистые функции).
- **`aggregate`** — агрегация по разрезу (View by) + общие итоги (чистые функции).
- **`api`** — методы для UI: получить текущий вид, выбрать файл, обновить, экспорт.
- **`main`** — запуск окна, загрузка дашборда, логика старта (см. §3).

Frontend (тонкий слой отрисовки):
- **`index.html` / `styles.css`** — разметка и стиль.
- **`ui.js`** — рендер таблицы/фильтров/итогов, переключатель разреза,
  сортировка и раскрытие строк, вызовы методов `api`.

**Технологии:** Python + `pywebview` (нативное окно с системным webview;
на Windows — Edge WebView2), `openpyxl` для чтения/записи `.xlsx`, упаковка в
один `.exe` через PyInstaller. Выбор обоснован переиспользованием Python-стека
(парсинг уже отработан) и простым доступом к ФС/диалогам.

**Распространение:** готовый `.exe` **самодостаточен** — интерпретатор Python и
все библиотеки упакованы внутрь, поэтому на машинах пользователей Python ставить
не нужно. Единственная внешняя зависимость — WebView2 (на Windows 11
предустановлен; при отсутствии — однократная тихая установка компонента MS).

## 8. Обработка ошибок и крайние случаи

- Сохранённый путь отсутствует/файл удалён или перемещён → диалог выбора файла с
  понятным пояснением.
- Файл не `.xlsx` / структура не распознана → сообщение, без сбоя.
- Пустые дни (дата есть, значений нет) — не дают строк.
- Несколько листов — берётся лист с распознанной структурой; пустые игнорируются.
- Пустой результат фильтра → пустая таблица, обнулённые итоги, понятная подпись.
- Файл занят (открыт в Excel) → читаем копию/сообщаем; не падаем.
- Иврит/RTL значений отображаются корректно (`dir="auto"`).

## 9. Допущения (подтвердить при ревью)

1. **Рабочая неделя — воскресенье–пятница** (суббота выходная). Используется для
   поля/фильтра `Weekday` и группировки по неделе (неделя начинается с вс).
2. Файл фиксирует **только опоздания**; длинная таблица = список случаев опозданий.
3. Опоздание в пределах (маршрут, день) одинаково; если различается — каждая
   строка-факт хранит своё значение (агрегаты суммируют).
4. «Человеко-минуты» = сумма `Late` по строкам среза.
5. Подписи интерфейса и заголовки — английский; значения (имена, города,
   маршруты) — иврит, как в данных.
6. Подрядчики-перевозчики в данных не выделены — анализ по маршрутам.
7. На целевой машине доступен Edge WebView2 (на Windows 11 обычно предустановлен).

## 10. Тестирование

- **Python:** юнит-тесты `excel_reader` (разбор формата, разворот в длинную
  таблицу), `filters`, `aggregate`, `config` на синтетических данных.
- **Контрольная проверка на реальном файле** по известным числам:
  - 01.06: Петах-Тиква — 28 случаев × 20 мин (= 560); Йехуд–Кирьят-Оно — 7 × 25
    (= 175);
  - 02.06: Лод — 11 × 22 (= 242); Йехуд–Кирьят-Оно — 7 × 20 (= 140).
  - Разрез View by Route без фильтра по дате даёт суммы этих величин.

## 11. Вне рамок (YAGNI / возможное развитие)

- Графики/диаграммы динамики (поверх той же длинной таблицы).
- Сопоставление маршрутов с подрядчиками (нужен отдельный справочник).
- Экспорт в формат зарплатной/учётной системы (нужен формат импорта).
- Сохранение пресетов фильтров между сессиями.
